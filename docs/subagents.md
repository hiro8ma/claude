# Claude Code Subagents

## サブエージェントとは

独立した Claude インスタンスで、各自のコンテキストウィンドウを持つ。メイン会話の履歴に影響されず、タスクを実行して結果のみを返す。

**コンテキストエンジニアリングの観点**: エージェント分割パターン（Ch5）の実装。コンテキストを分離して別エージェントに任せることで、メイン会話の肥大化を防ぐ。

## 使うべき4つのケース

### 1. リサーチタスク
- 数十ファイルの読み込みが必要な場合
- 生データの代わりに要約が返される → メイン会話がクリーン

### 2. 独立した複数タスク
- サブタスク間に依存関係がない場合
- 並列実行で処理時間が短縮

### 3. 客観的なレビュー
- 検証が必要で会話履歴の影響を受けたくない場合
- 前提条件や盲点を持たないため客観的なフィードバック

### 4. パイプラインワークフロー
- 設計→実装→テストのような段階的タスク
- 各段階が独立して集中できる

## 呼び出し方法5つ

| 方法 | 定義場所 | ユースケース |
|---|---|---|
| **会話的呼び出し** | その場で自然言語指示 | 最も柔軟。まずこれから始める |
| **カスタムサブエージェント** | `.claude/agents/*.md` | 頻繁に使う専門的な役割 |
| **CLAUDE.md** | `CLAUDE.md` | プロジェクト全体のルール |
| **スキル** | `.claude/skills/*/SKILL.md` | 複雑なマルチステップワークフロー |
| **フック** | `settings.json` | イベント時に自動実行（コミット前レビュー等） |

## 使うべきでない場合

- 順序依存の作業（ステップ2がステップ1の完全な出力を必要）
- 同一ファイルの並列編集
- 小規模な修正や簡単な質問
- サブエージェント間の相互調整が必要 → Agent Teams を検討

## ベストプラクティス

### 段階的な立ち上げ

> 会話的呼び出しから始め、繰り返されるパターンが見えたら自動化を導入。最初からすべてを自動化しない。

### 明確なスコープ指定

- 「漠然と探索」より「認証フロー図を作成」
- 出力形式も「優先度付きリスト」と指定

### 実装前のリサーチ

複数の調査を並列実行させ、要約結果を受け取ってから実装に入る。

## MCP / Skills / サブエージェントの使い分け

| 方式 | コンテキスト | 適するタスク |
|---|---|---|
| **MCP（ツール）** | 常時含める | シンプルな処理 |
| **Skills** | 段階的にロード | 中程度の複雑さ |
| **サブエージェント** | 分離 | 重厚なタスク、リサーチ、レビュー |
